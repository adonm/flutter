# Linux GTK Variant Selection

Flutter's Linux desktop embedder supports GTK3 and GTK4 as separate native
runner libraries:

- `libflutter_linux_gtk.so` links GTK3.
- `libflutter_linux_gtk4.so` links GTK4.

GTK3 remains the default. GTK3 and GTK4 cannot be loaded into one process, so
every native Linux plugin in an application must compile against the selected
GTK major.

## Selecting a GTK version

For a project-wide default, set `flutter.config.linux-gtk-default` in
`pubspec.yaml`:

```yaml
flutter:
  config:
    linux-gtk-default: gtk4
```

A command-line value overrides the project default for one invocation:

```bash
flutter build linux --linux-gtk=gtk4
flutter run --linux-gtk=gtk4
```

A user-wide default can be set with:

```bash
flutter config --linux-gtk-default=gtk4
```

The Flutter tool resolves the version in this order:

1. `--linux-gtk`
2. `flutter.config.linux-gtk-default`
3. `flutter config --linux-gtk-default`
4. `gtk3`

The resolved value controls the engine artifact, Dart build define, CMake
configuration, and output directory. `FLUTTER_LINUX_GTK` is an internal build
define produced by the Flutter tool; it is not a supported user-facing
selector. When invoking generated CMake projects without the Flutter tool,
pass `-DLINUX_GTK_VARIANT=gtk3` or `-DLINUX_GTK_VARIANT=gtk4`.

GTK4 output is kept separate from GTK3 output:

```text
build/linux/x64/release/...
build/linux-gtk4/x64/release/...
```

## Generated runner

New Linux projects use one runner source tree that contains guarded GTK3 and
GTK4 code. The project default determines which path is compiled, so the same
checked-in `linux/` directory can be built with either command-line value.

Projects generated by Flutter versions that predate the unified runner remain
valid GTK3 projects. Before selecting GTK4, regenerate the Linux runner or port
application-specific runner changes onto a newly generated project:

```bash
flutter create --platforms=linux --linux-gtk=gtk4 gtk4_reference
```

The Flutter tool fails with migration guidance rather than combining a GTK4
engine artifact with a GTK3-only runner.

## GTK4 windowing limitations

Regular and dialog windows support creation, resizing, minimization,
maximization, fullscreen state, activation, titles, and lifecycle callbacks.
GTK4 does not provide an equivalent for GTK3 maximum geometry hints, so finite
maximum constraints throw `UnsupportedError`; minimum-only constraints remain
supported. Popup and tooltip windows are not yet implemented for GTK4.

As with GTK3, a compositor can decline resize, minimize, maximize, or
fullscreen requests. Wayland compositors also do not report minimized state.

## Plugin compatibility contract

Generated projects expose the selected major through the `flutter` CMake
interface target. Native plugin targets inherit the definition by linking that
target:

```cmake
target_link_libraries(my_plugin PRIVATE flutter)
target_link_libraries(my_plugin PRIVATE PkgConfig::GTK)
```

Dual-compatible plugin source should branch only on the positive GTK4 marker:

```cpp
#if defined(FLUTTER_LINUX_GTK4)
// GTK4 implementation.
#else
// GTK3 implementation. An undefined marker means an older GTK3 Flutter SDK.
#endif
```

New runners also define `FLUTTER_LINUX_GTK3` for GTK3 builds, but plugins must
not require it: existing Flutter projects do not export that marker. This
fallback allows one plugin source release to support older GTK3 Flutter SDKs
and GTK4-capable SDKs. Each application build still produces a library for one
GTK major only.

For downstream plugin migration and application acceptance requirements, see
[Linux GTK4 Downstream Compatibility](linux-gtk4-downstream.md).
